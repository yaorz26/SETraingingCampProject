# REFLECTION.md

做这个项目之前，我对"AI 写代码"的理解还停留在让 ChatGPT 帮我补全函数的程度。一个学期下来，最大的变化是我开始意识到：LLM 本身不是产品，围绕它搭的那层工程才是。这个道理课程里反复讲，但只有自己亲手把 agent 主循环、护栏、反馈闭环一行行写出来，才真正体会到它的分量。

## Superpowers 哪些有用、哪些形式大于实质

brainstorming 技能是整个流程里对我帮助最大的。它不是简单地问"你想做什么"，而是会追问到很细的地方——比如我一开始说"要做护栏"，它追问"护栏分几层、每层拦截什么、fatal 和 dangerous 的区别在哪、会话白名单要不要持久化"。这些问题我自己根本想不到去问自己。三轮 brainstorming 下来，SPEC 从一个模糊想法变成了 1800 多行的文档，后来实现的时候几乎没有遇到"设计没想清楚"导致的返工。

writing-plans 也比较实在，把设计拆成 44 个 task，每个都标了涉及文件和验证步骤。但说实话，有些 task 的颗粒度不太均匀。T0.1 这种脚手架 task 写了十几条实现要点，实际十分钟就搞完了；而 T3.1 动作类型定义看起来一条，实际写了快两百行代码。颗粒度不均匀倒没造成大问题，就是有时候会误判某个 task 要花多久。

requesting-code-review 这个技能我觉得作用一般。它生成的评审意见大部分是"建议补充类型注释""变量命名可以更清晰"这类话，真正抓到逻辑问题的次数不多。可能是因为我写的代码本身比较简单，也可能是 review 技能对业务逻辑的理解不够深。相比之下，我自己跑测试、看覆盖率反而更直接。

## TDD 在 AI 协作下是阻碍还是放大器

一开始我觉得 TDD 很烦。明明知道要写什么，为什么要先写一个注定失败的测试？但做到后面我理解了——TDD 的价值不在于"测试驱动设计"那套理论，而在于它给 AI 设了一个锚点。我让 subagent 写 action-parser 的时候，如果只给它 SPEC 描述，它写出来的东西经常多一堆我没要求的功能。但如果我先写好失败的测试用例，告诉它"让这些测试通过，别的不许加"，它的输出就收敛很多。

所以 TDD 在这个项目里更像是约束 AI 的工具，而不是传统意义上的开发方法论。先写测试的本质是：你被迫在写代码之前就把"完成"的标准定义清楚。这个标准是客观的，AI 不能靠"看起来对了"蒙混过关。

不过也有反例。drift-detector 那块我先写了测试，但测试本身写得不好——只测了正常路径，没测边界情况。结果 subagent 照着测试写了个很简陋的实现，后来我自己补了好几个 case 才把逻辑补全。这说明 TDD 的效果取决于测试质量，垃圾测试只会产出垃圾实现。

## subagent 能自主跑多久不跑偏

我的经验是，单个 task 颗粒度下，subagent 基本能自己跑完不偏题。但一旦 task 描述含糊，或者涉及多个文件的关联修改，它就开始"自由发挥"。最典型的一次是让 subagent 实现 file-ops 的原子写入，SPEC 里写了"临时文件 + rename"策略，但它自作主张加了一个文件锁机制，理由是"防止并发写入"。这个功能我没要求，而且引入了平台兼容性问题。后来我在 prompt 里明确加了"只实现 SPEC 描述的逻辑，不要添加额外功能"，这类情况就少了很多。

所以自主运行时长跟 prompt 清晰度强相关。我试过让 subagent 连续做三个关联 task，中间不干预，结果第三个 task 就开始偏离了——它把前一个 task 的实现细节当成了通用约定，到处套用。从那以后我基本一个 task 一个 session，做完就重新开。

## SPEC 质量怎么影响实现

有个很具体的例子。SPEC §3.5.1 列了 17 种工具，但实现的时候我发现 action-parser 只定义了 12 种——git_status、git_diff 这些被合并进了 run_command。这个差异不是设计变更，而是我写 SPEC 的时候没想清楚"工具"和"动作"的边界。git_status 到底是一个独立动作，还是 run_command 的一种用法？SPEC 里两边都说了，自相矛盾。subagent 实现的时候直接按 12 种做了，因为它读到的 PLAN task 写的是 12 种。后来我索性把 SPEC 也改成 12 种，保持一致。

这件事让我意识到，SPEC 不是写完就定型的东西。实现过程中会暴露设计里没想清楚的点，这时候要回去改 SPEC，而不是假装没看见。但改 SPEC 又要同步改 PLAN，工作量不小。我最后没做到"每次实现偏差都同步回 SPEC"，这是做得不够的地方。

## 凭据和分发逼我想清楚的事

凭据这块一开始我觉得很简单，存个 API Key 嘛。真做的时候才发现问题一堆：Key 存哪、怎么加密、测试时怎么 mock、日志里会不会泄露、首次运行怎么引导用户输入。SPEC 里写"使用 OS 钥匙串"，但 Windows 上 keytar 的行为跟 macOS 不一样，测试环境又没有钥匙串。最后我改成了 AES-256-GCM 加密文件存储，虽然不如 OS 钥匙串安全，但跨平台一致、可测试。这个妥协是我权衡后的决定，SPEC 里也如实写了明文风险。

分发逼我想的是"别人拿到我的代码后怎么跑起来"。Dockerfile 写起来不难，但要想清楚容器里怎么配置 Key、工作目录怎么挂载、非 root 用户怎么跑。这些平时写 demo 根本不会考虑的问题，一旦要"交付给别人"就躲不掉。

## 如果重做我会改什么

第一，我会把 SPEC 写得更克制。1800 行的 SPEC 信息量很大，但很多细节在实现时根本没用到，反而增加了阅读负担。比如 §3.8 的终端输出示例，写得很详细，但实际 CLI 输出是另一套实现，SPEC 里的示例更像是给自己画的设计稿，对 subagent 没什么指导意义。

第二，我会更早地做集成测试。单元测试写了很多，但模块之间的衔接问题要等集成测试才暴露。比如 action-dispatcher 单测全过，但接进 agent-loop 后才发现路径处理有个边界 bug。如果早点写 e2e 测试，能省不少调试时间。

第三，commit 历史应该更规范。我最后基本是一个大 commit 提交的，这不符合课程要求，也不利于回溯。应该每个 task 完成就 commit，附上 task 编号。

## 对 Superpowers 的批判

这套方法论有一个核心假设：spec 足够清晰，subagent 就能自主推进。在我的项目里，这个假设大部分时候成立，但成立的条件是 spec 的每个细节都经过推敲。而推敲细节的活，框架替不了你做。brainstorming 会追问，但追问的质量取决于你对问题的理解深度。如果我自己都没想清楚，brainstorming 问出来的答案也是含糊的。

另一个假设是"先 spec 后实现"的顺序严格优于边做边想。我的体验是，有些设计问题只有写了代码才暴露，强行在 spec 阶段定死反而会写出脱离实际的规约。也许 spec 和实现应该有更多轮的来回，而不是一次 brainstorming 冻结后就尽量不改。

总的来说，这个项目让我对"工程师在 AI 时代的价值"有了具体认识。价值不在写代码——那部分 LLM 确实能做。价值在于定义"做什么"、判断"做对了吗"、以及搭出让 AI 可靠运转的那层工程。这层工程不性感，但缺了它，LLM 就只是一个不可控的黑盒。
