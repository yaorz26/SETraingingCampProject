## 2. 第一轮：项目定位与技术选型

### 2.1 AI 提出的关键问题与我的决策

#### Q1: 目标用户是谁？使用场景是什么？

**AI 追问**：是给个人开发者用的 CLI 还是给其他系统集成的 SDK？面向什么规模的编码任务？

**我的决策**：**两者兼顾**——CLI 是主要交互形态，但内部架构分层清晰，核心 harness 模块可作为 SDK 被其他系统集成。面向"单文件修改到中等规模多文件重构"，不追求全自动开发整个项目——这一定位让"治理护栏"和"反馈闭环"的工程价值最容易体现。

**此决策修正了我的原设想**：我最初想做一个"全自动 AI 程序员"，但 AI 追问让我意识到边界明确、可控的定位反而更能体现 harness 的核心价值。

---

#### Q3: 编程语言选什么？

**AI 分析**：列出 TypeScript/Node.js、Python、Go、Rust 四种选项的优劣对比表，追问我的选择及理由。

**我的决策**：**TypeScript / Node.js**。理由：
1. 本次课程宿主生态就是 Node.js/TypeScript，同构减少环境摩擦
2. CLI 工具生态成熟（commander、inquirer、chalk、ora）
3. 静态类型在"动作分发"系统中价值巨大
4. LLM SDK 丰富（OpenAI、Anthropic、Ollama 均有官方或高质量社区 SDK）

**此决策是 AI 引导我系统化思考的结果**：AI 没有直接推荐，而是列出对比表让我自己权衡。我最初倾向 Python（AI 生态最成熟），但经过对比后认识到 TypeScript 在"类型安全 + CLI 工具链 + 与课程环境同构"三个维度上的综合优势。

---

#### Q4: 对接哪些 LLM 供应商？

**AI 追问**：只对接一个还是多个？是否支持本地模型？API Key 由谁提供？

**我的决策**：
- 支持多个供应商（OpenAI + Anthropic + Ollama）
- 通过统一 `LLMProvider` 接口实现可替换
- API Key 完全由用户提供，程序内置凭据引导流程
- MVP 阶段先对接 OpenAI 和 Anthropic，Ollama 作为扩展

**此决策被后续轮次深化**：第二轮确认了 Adapter 模式 + 强制 Tool Use 的策略；第三轮确认了降级链机制。

---

#### Q5-Q6: 工具范围与危险动作

**AI 追问**：逐一确认 9 种操作是否纳入，并要求列出危险动作清单和审批交互方式。

**我的决策**：
- 纳入：文件读写/删除、Shell 执行、测试/Lint/类型检查/Build 运行
- 部分纳入：Git 只读操作（如 `git status`、`git diff`）允许，写操作（`commit`、`push`）需审批；网络请求仅限包管理命令
- 危险动作清单：8 类（文件破坏、Git 危险操作、权限变更、任意代码执行、数据库破坏、系统级操作、敏感文件访问、对外发布）
- 审批交互：CLI 输入 `y/n/a/s`，支持"本次会话记住"（`A` 选项），但不支持"永久白名单"

**此决策在第二轮被大幅深化**：从简单的"危险模式列表"深化为五层护栏架构（L1-L5），并明确了各层的拦截顺序和审批状态机。

---

#### Q8: 重点深入维度

**AI 分析**：建议三个可做深的维度——治理（护栏）、反馈闭环、记忆/上下文工程，追问我的选择及理由。

**我的决策**：**治理（护栏/沙箱/HITL 状态机/范围围栏）**。理由：
1. 最"代码化"的维度——天然契合"移除 LLM 还能单测"的硬标准
2. 工程深度最好体现——五层架构（模式匹配→路径围栏→权限分级→HITL→审计日志）
3. 机制演示最容易做——三个演示中有两个直接展示治理能力
4. 与其他维度自然关联——治理深入会自然触及反馈、配置、工具分发

**此决策是项目最核心的方向性决策之一**。AI 提出的三个候选维度都很有深度，但我选择治理是因为它最能体现"机制是代码，不是提示词"这一核心原则。

---

#### Q9-Q10: 交互模式与配置

**AI 追问**：交互式还是非交互式？配置方式优先级？

**我的决策**：
- 两者都支持，通过 CLI 参数切换（`--interactive` / `--non-interactive`）
- 多层配置体系：CLI 参数 > 环境变量 > 项目级 YAML > 全局 YAML > 内置默认值
- 配置文件格式：YAML（比 JSON 更易读写，比 TOML 更主流）

**此决策在第二轮被修正**：配置文件名称从 `.agent.yaml` 统一为 `.codeharness.yaml`；合并策略从"未明确"明确为"深度合并"。

---

#### Q11: 分发形态

**AI 分析**：列出容器镜像、原生二进制、包管理器三种形态，追问目标平台。

**我的决策**：**主选 npm 包，次选原生二进制**。不选容器镜像（CLI 工具用容器分发 UX 较差）。目标平台全平台（Linux/macOS/Windows）。

---

#### Q13: 风险预判

**AI 追问**：预见到哪些可能让开发出问题的环节？

**我识别了 6 大风险**：
1. LLM 输出格式不稳定（对策：强制 Tool Use + 重试机制）
2. Shell 命令安全沙箱（对策：多层护栏 + 子进程隔离 + 明确文档说明安全边界）
3. Mock LLM 设计（对策：基于场景的响应序列，约 50 行极简实现）
4. 上下文窗口有限（对策：分层上下文管理 + 按 token 数裁剪）
5. 跨平台兼容性（对策：自动检测平台 + path 模块统一）
6. Agent 无限修正循环（对策：`auto_fix_rounds` 硬上限 + 连续相同 diff 检测）

**AI 没有对风险预判提出异议，但第二轮和第三轮针对每个风险都做了深化设计**。

## 3. 第二轮：核心机制深度设计

### 3.1 AI 提出的关键问题与我的决策

#### Q2.1: 命名统一

**AI 追问**：第一轮中存在命名不一致（`my-harness` vs `codeharness`、`.agent.yaml` vs `.codeharness.yaml`），要求统一。

**我的决策**：全面统一为 `codeharness` 前缀——包名 `codeharness`、CLI 命令 `codeharness`（别名 `ch`）、全局配置目录 `~/.codeharness/`、项目配置文件 `.codeharness.yaml`、记忆文件 `.codeharness/memory.json`。

**这是纯"AI 发现我的不一致并指出"**——我自己没有意识到这些命名混乱。

---

#### Q2.2: HITL 状态机

**AI 追问**（非常深入）：
- 拒绝（DENIED）状态是否存在？Agent 收到拒绝后进入什么状态？
- 超时是独立状态还是子状态？
- "同类操作"如何定义？按危险模式类别还是精确命令匹配？
- 会话白名单存储在哪里？

**我的决策**：
- DENIED 和 TIMED_OUT 都是**独立状态**
- 拒绝后注入结构化反馈到 LLM 上下文，Agent 据此调整策略
- "同类"按**危险模式类别**（`DangerCategory` 枚举，7 个类别），而非精确命令匹配
- 会话白名单仅存**内存中**（`Map<DangerCategory, boolean>`），不落盘——"会话级信任不应跨会话保留"

**这是第二轮中 AI 追问最深入、让我思考最多的一个问题**。在 AI 追问之前，我只有模糊的"IDLE→EXECUTING→AWAITING→APPROVED→IDLE"线性状态；AI 追问后，我画出了完整的分支状态图，明确了 DENIED、TIMED_OUT、BLOCKED 三种终止路径的不同语义。

---

#### Q2.3: 护栏执行顺序

**AI 追问**：给出 L1→L2→L3→L4→L5 流水线草案，问我是否正确。

**我的决策**：基本正确，但做了关键修正：
- L1（模式匹配）和 L2（路径围栏）是**硬拦截**（不可审批），命中直接 BLOCKED
- L3（权限分级）不是独立审批环节，而是**风险评级**，为 L4 提供决策依据
- L4（HITL）只在 `caution`（首次）和 `dangerous/critical` 级别触发
- L5（审计日志）**始终执行**（无论通过还是拒绝）

**AI 的草案被修正了**：AI 原草案将 L3 作为"另一层审批"，我修正为"风险评级 + 条件触发"。

---

#### Q2.4: 反馈收集的执行顺序

**AI 追问**：5 种反馈信号是否总是全部运行？还是按优先级/依赖关系选择性运行？回灌是逐条给还是一次性给？

**我的决策**：
- 按依赖关系**选择性运行**，引入**依赖短路**规则：
  - 类型检查失败 → 跳过构建
  - 构建失败 → 跳过测试
  - Lint 和类型检查覆盖不同问题，可并行
- 所有结果汇总为一条结构化反馈，**一次性回灌**（而非逐条给）——让 Agent 看到全景，做出更合理的修正决策

**AI 的追问直接催生了"依赖短路"这个设计**。在 AI 追问之前，我默认"全部运行"，没有考虑构建失败后测试无意义的浪费。

---

#### Q2.5: Diff 反馈信号

**AI 追问**：是对比 git diff 还是快照？"预期"如何判定？

**我的决策**：
- 使用**快照 hash 对比**（而非 git diff），因为 git diff 对新建文件不适用
- "预期"判定：Agent 声明的目标文件 vs 实际修改文件取差集
- 意外修改 = 实际文件 - 预期文件

**这是我之前完全没有想过的细节**。AI 追问让我意识到"Diff 反馈"不是简单调用 `git diff`，而是需要设计独立的快照追踪器。

---

#### Q2.6: Workspace Root 检测

**AI 追问**：如何确定工作区根目录？如果用户在子目录下运行怎么办？

**我的决策**：四级检测策略——
1. 显式参数（CLI/配置）
2. 环境变量 `CODEHARNESS_WORKSPACE`
3. 向上查找项目标记（`.git`、`package.json`、`Cargo.toml`、`go.mod`）
4. 默认 `process.cwd()`

**AI 的追问场景非常具体**（"用户在 `/frontend/` 下运行，但根目录是 `/myapp/`"），让我必须给出精确的查找逻辑。

---

#### Q2.7: 网络请求边界

**AI 追问**：5 个具体场景——`npm install` 含 `postinstall` 恶意脚本、`wget + chmod + ./script.sh`、`git clone` 外部仓库、`npx` 远程包、`pip install --user`。

**我的决策**：
- 逐一给出处理方式
- 明确声明**不可防御的风险**（包管理器的恶意依赖、LLM 提示注入、零日漏洞）
- 文档中写入风险声明："CodeHarness 的护栏是防御性编程，不是安全沙箱"

**AI 追问的这 5 个场景非常刁钻**，让我意识到护栏的边界——有些风险（如 `postinstall` 脚本）是包管理器自身的机制，harness 层面无法完全防御，必须诚实声明。

---

#### Q2.8: 文件操作原子性

**AI 追问**：修改文件时中途出错如何保证不留下半成品？

**我的决策**：
- 写临时文件（`*.codeharness-tmp-<uuid>`）→ 验证完整性 → 备份原文件 → 原子 rename
- 修改前自动备份到 `.codeharness/backups/`
- 错误时清理临时文件，回滚从备份恢复
- 任务成功完成 → 清理所有备份；任务中断 → 下次启动提示恢复

**这是 AI 提出而我完全采纳的设计**。在 AI 追问之前，我默认使用 `fs.writeFile` 直接覆写，没有考虑原子性。AI 追问后，我设计了完整的"临时文件 + rename + 备份 + 回滚"机制。

---

#### Q2.9: 多供应商 API 差异

**AI 追问**：列出 OpenAI/Anthropic/Ollama 在消息格式、Tool Use、JSON Mode、System Prompt 四个维度的差异，追问如何统一。

**我的决策**：
- 采用 **Adapter 模式**，在 `LLMProvider` 接口之下增加格式适配层
- **强制使用 Tool Use / Function Calling** 作为动作输出方式——不支持 tool use 的供应商不予支持
- 放弃 JSON Mode 方案（Anthropic 不支持，解析脆弱）
- 放弃提示词约束方案（不可靠，违反"机制必须是代码"原则）

**此决策是第二轮中最重要的架构决策之一**。AI 没有直接推荐方案，而是列出差异表让我面对"强制 tool use vs JSON Mode vs 提示词约束"的选择困境，最终我选择了最可靠但最严格的方案。

---

#### Q2.10: 动作 Schema

**AI 追问**：给出不完整的动作类型定义草稿，要求补充完整。

**我的决策**：定义了 12 种完整动作类型——
- 文件操作：`read_file`、`write_file`、`delete_file`、`list_dir`、`search_file`、`grep`
- 命令执行：`run_command`
- 反馈触发：`run_tests`、`run_lint`、`run_type_check`
- 控制流：`finish`、`ask_user`

同时还定义了 `ActionResult` 和 `GuardrailResult` 的完整类型。

**AI 的草稿只列了 5 种**，我补充了 7 种。这是典型的"AI 给框架，人类填细节"的协作模式。

---

#### Q2.11: 跨会话记忆

**AI 追问**：列出 6 种可能的记忆内容，逐项询问是否纳入及理由。

**我的决策**：
- ✅ 纳入：项目文件结构摘要、历史任务记录（最近 10 次）、用户偏好、项目约定
- ❌ 排除：已批准的护栏规则（跨会话）（安全考量）、常见错误模式（MVP 不做，工程投入过大）

**特别重要的决策**：**跨会话不保留护栏白名单**——"每次新会话重新建立信任"。这是安全维度的审慎决策。

---

#### Q2.12: 上下文裁剪策略

**AI 追问**：阈值是多少？裁剪优先级？按轮数还是按 token 数？

**我的决策**：
- 阈值：80% 模型上下文窗口（保留 20% 余量给 LLM 输出）
- 裁剪优先级（从先丢弃到后丢弃）：旧对话轮次 → 深层目录细节 → 非当前任务代码 → 历史任务详细描述 → 永不丢弃（系统提示、当前反馈、最近 2 轮）
- **按 token 数裁剪，非按轮数**——某轮包含大量代码时按轮数不准确

**AI 追问的"按 token 数而非轮数"是核心洞察**。如果按轮数裁剪，一轮包含 2000 行代码和一轮只有一句话被等同对待，显然不合理。

---

#### Q2.13: 启动失败场景

**AI 追问**：列出 6 种启动失败场景，要求确定每种的行为和退出码。

**我的决策**：定义了 10 种统一退出码（0-9），每种失败场景对应明确的退出码和用户提示。关键原则：**配置解析失败不静默回退到默认值**（避免用户以为配置生效了但实际被忽略）。

---

#### Q2.16: 机制演示

**AI 追问**：要求为三个演示设计完整的 mock LLM 响应序列。

**我的决策**：设计了三个完整的演示脚本，每个包含：
- 演示①（护栏拦截）：mock LLM 输出 `rm -rf /tmp/...` → 护栏拦截 → 模拟用户审批 → 继续
- 演示②（反馈修正）：mock LLM 故意输出错误代码（`a - b`）→ 测试失败 → LLM 收到反馈 → 修正为 `a + b` → 测试通过
- 演示③（治理深度）：四个场景（正常通过、危险审批、越界拒绝、同类别记忆）的完整组合

**AI 追问让我意识到**：演示不是"跑一遍看效果"，而是需要**预先设计 mock LLM 的每一步响应**，确保演示的确定性——每次运行结果完全一致。

### 3.2 第二轮小结

| 形成的决策 | AI 追问的价值 |
|-----------|-------------|
| 命名全面统一为 `codeharness` | AI 发现了我第一轮中的多处不一致 |
| 完整 HITL 状态图（含 DENIED/TIMED_OUT/BLOCKED） | AI 追问"拒绝后 Agent 进入什么状态"催生了完整状态图 |
| 五层护栏执行顺序（L1/L2 硬拦截，L3 评级，L4 条件触发，L5 始终） | 修正了 AI 草案中 L3 作为"另一层审批"的错误 |
| 反馈收集依赖短路 + 一次性回灌 | AI 追问"构建失败还需要测试吗"催生了短路设计 |
| 文件操作原子性（临时文件 + rename + 备份） | AI 提出，我完全采纳 |
| 强制 Tool Use + Adapter 模式 | AI 列出差异表，我做出选择 |
| 12 种完整动作类型 | AI 给框架，我填细节 |
| 上下文按 token 数裁剪 | AI 追问的核心洞察 |

**第二轮让我修正的原设想**：
- 护栏从"列表匹配"→"五层架构"（L3 角色的修正尤其关键）
- 反馈收集从"全部运行"→"依赖短路"（节省无意义的时间和成本）
- 文件操作从"直接覆写"→"原子写入"（工程安全性大幅提升）
- LLM 适配从"可选 tool use"→"强制 tool use"（最严格的方案，但最可靠）

## 4. 第三轮：工程实践与边界情况

### 4.1 AI 提出的关键问题与我的决策

#### Q3.1: 系统提示设计

**AI 追问**：要求给出完整的系统提示草稿框架，包含角色定义、工具列表、行为约束、反馈处理规则、禁止事项。

**我的决策**：撰写了约 700 字的系统提示草稿，核心设计原则：
1. **角色精准**："严谨、务实的软件工程师"而非"超级 AI 程序员"——降低 LLM 过度自信
2. **工具表格式描述**：让 LLM 清楚每个工具何时使用
3. **自主 vs 询问的明确边界**：列出 5 种自主决策场景和 5 种必须询问场景
4. **反馈处理规则具体**：每种反馈类型有明确的应对策略
5. **禁止事项硬约束**：9 条禁止事项，配合代码层护栏形成双重防护

**这是之前两轮完全没有触及的领域**。AI 追问让我意识到系统提示不是"随便写几句"，而是需要精心设计——它决定了 Agent 的"人格"和行为质量。

---

#### Q3.2: 目标漂移防护

**AI 追问**：LLM 在多轮任务中容易"忘记最初目标"，如何防止？偏离检测由谁来做——代码还是 LLM 自检？

**我的决策**：三层防护机制——
1. **上下文注入**（每轮重复原始任务描述）
2. **偏离检测**（代码逻辑，非 LLM 自检）：检测修改范围暴增、文件与任务无关、修改配置文件但任务不涉及配置
3. **偏离响应**：低风险提醒，高风险拦截并询问用户

**关键决策**：偏离检测必须由**代码逻辑**实现，而非 LLM 自检。原因：
- LLM 自检不可靠（它可能为自己的偏离找理由）
- 代码逻辑是确定性的，遵循"移除 LLM 后仍可单测"原则
- 偏离检测的规则是启发式的（明确标注），在 harness 代码中透明可见

**AI 追问的"谁来做偏离检测"是关键问题**。如果回答"让 LLM 自己检查自己有没有跑偏"，那就违反了"机制必须是代码"的核心原则。

---

#### Q3.3: PLAN.md 执行策略

**AI 追问**：哪些 task 可并行？冒烟测试？核心机制开发顺序？

**我的决策**：
- 可并行组：T1+T3（配置+LLM）、T2+T4（日志+上下文）、T5+T7（护栏+反馈）
- 必须串行：T0 → T1/T2/T3/T4 → T5+T7 → T6 → T8 → T9 → T10
- T0 完成后立刻写冒烟测试（mock LLM → 主循环 → finish），验证整个骨架
- **逐个实现并集成**（非全部实现再集成）——早期集成暴露接口设计问题

---

#### Q3.4: 多语言项目支持

**AI 追问**：5 个场景——Python 项目、不同测试命令、非 Git 项目、Monorepo、Makefile 项目。

**我的决策**：
- 核心原则：**CodeHarness 完全语言无关**——只操作文件、执行命令、解析退出码
- 所有语言特定的东西在 `.codeharness.yaml` 中配置
- Monorepo **不自动处理**（MVP 不支持），用户需在子项目目录中放置各自配置

---

#### Q3.6: 流式输出与用户体验

**AI 追问**：LLM 思考过程是否展示？执行前是否展示计划？是否实时展示命令输出？是否有 verbose 模式？

**我的决策**：
- **不展示 LLM 思考流**（raw token 流对用户无意义），显示 spinner + 状态文本
- 展示**单行动作摘要**（非多步计划，不需要用户确认——危险操作已有审批机会）
- 命令执行时**实时展示 stdout/stderr**（交互模式），非交互模式抑制输出
- 支持 `--verbose` 模式（展示完整 LLM 请求/响应、上下文大小、护栏详情、每步耗时）

---

#### Q3.7: Dry-Run 模式

**AI 追问**：是否支持"仅规划不执行"？dry-run 中是否允许读文件？是否消耗 LLM 调用？

**我的决策**：支持 `--dry-run`。Dry-run 中：
- ✅ 允许读文件/搜索（只读操作）
- ❌ 禁止写文件/执行命令（记录到计划中）
- ✅ 会消耗 LLM 调用（需要 LLM 分析代码并生成计划）
- 实现方式：给主循环传递 `dryRun: true` 标志，工具分发层在写操作前检查

---

#### Q3.8: 任务完成的判定

**AI 追问**：Agent 调用 `finish { success: true }` 但测试未通过，harness 是否拦截？Agent 宣告失败时用户看到什么？用户能否强制结束？

**我的决策**：**三层判定机制**——
1. **Agent 调用 finish 时 harness 拦截**：如果测试未通过或有意外修改，harness 拦截 `finish` 动作，将矛盾信息回灌 LLM
2. **Agent 宣告失败**：展示失败原因 + 已修改文件列表 + 建议 + 回滚命令
3. **用户强制结束**：达到最大轮数或 `Ctrl+C` 两次，询问是否保留已修改文件

**核心原则**：**harness 不盲目信任 Agent 的 `finish`**。如果客观反馈信号表明任务未完成，harness 会拦截并回灌。

---

#### Q3.9-Q3.11: LLM 调用策略

**AI 追问**：速率限制重试？Token 成本追踪？模型降级？

**我的决策**：
- **重试**：最多 3 次，指数退避 + 抖动，优先使用 `Retry-After` 响应头
- **成本追踪**：任务后展示 token 用量和估算成本，支持 `--max-cost` 预算上限，审计日志记录每次调用
- **降级链**：支持配置降级链（如 OpenAI → Anthropic → Ollama），4xx 错误不降级，降级后不回升

---

#### Q3.12-Q3.13: 测试策略

**AI 追问**：单元/集成/E2E/跨平台/性能/安全测试？覆盖率目标？哪些文件豁免？

**我的决策**：
- 单元测试（mock LLM）+ 集成测试（完整流程）+ CLI E2E（最小集）+ CI 跨平台矩阵
- 覆盖率 ≥ 80%（核心模块），豁免 CLI 入口、交互式向导、纯类型定义
- CI 强制覆盖率检查

---

#### Q3.14-Q3.15: 可观测性

**AI 追问**：审计/运行/调试/LLM 调用日志？日志格式？轮转策略？进度指示？

**我的决策**：
- 四类日志：审计（JSONL，始终记录）、运行（INFO，默认）、调试（DEBUG，`--log-level debug`）、LLM 调用（合入调试日志）
- 日志格式：JSONL（每行一个 JSON 对象）
- 轮转：单文件最大 10MB，保留最近 7 天
- 进度指示：轮次 + 动作状态图标 + 每轮耗时 + spinner

---

#### Q3.16: 首次运行体验

**AI 追问**：要求设计完整的首次运行向导流程。

**我的决策**：9 步向导——
1. 检测未配置 → 触发向导
2. 检测环境变量中的 Key
3. 引导选择供应商
4. 隐藏输入 API Key
5. 存储到 OS 凭据管理器
6. 验证 Key 有效性
7. 生成默认配置文件
8. 展示快速入门
9. 询问是否继续执行原始任务

**特别设计**：验证失败时提供"重新输入/更换供应商/退出"三个选项，而非直接退出。

#### Q3.18: 超大项目处理

**AI 追问**：10 万文件、50MB 单文件、grep 10 万行结果、50GB node_modules。

**我的决策**：
- `list_dir` 硬限制 500 条，`search_file` 硬限制 200 条，`grep` 硬限制 500 行
- 单文件超过 1MB 拒绝全量读取，提示使用分段读取
- 默认排除 11 个常见依赖目录（`node_modules`、`.git`、`dist`、`build` 等）

## 8. Brainstorming 方法论反思

### 8.1 做得好的地方

1. **多轮递进结构非常有效**：第一轮定方向，第二轮抠细节，第三轮封边界——每轮有明确的聚焦领域，不会在早期陷入细节，也不会在后期才发现方向问题。

2. **AI 的追问质量高**：AI 不是简单地"你决定"，而是给出对比表、列出具体场景、追问"如果……怎么办"。这种苏格拉底式的追问迫使我把模糊的直觉转化为精确的设计。

3. **"机制必须是代码"成为贯穿始终的试金石**：每次做设计决策时，我都会问自己"这个能用 mock LLM 做确定性单测吗？"——如果不能，就说明设计有问题。这个原则在第二轮和第三轮中反复出现，帮助我多次纠正了方向。

4. **51 个问题覆盖了全维度**：从技术选型到用户体验，从核心机制到边界情况，从开发策略到分发运维——没有明显的盲区。

5. **第二轮是我认为最有价值的一轮**：HITL 状态机、反馈短路、文件原子性、强制 Tool Use——这些决策直接决定了架构质量。第一轮不深究，第三轮不后悔。

### 8.2 让我不满的地方

1. **Brainstorming 与 SPEC 撰写之间的断层**：三轮 brainstorming 产生了大量决策，但要将其整理成结构化的 SPEC.md 仍然需要大量的人工整理工作。AI 没有自动将 brainstorming 对话转化为 SPEC 章节的能力。

2. **有些问题 AI 追问了两次**：例如命名统一（Q2.1）和配置文件格式（Q2.15），这些应该在第一轮就一次问清楚，而不是在第二轮才纠正。

3. **第三轮的问题数量偏多**（22 个），其中部分问题（如 Q3.5 开发工具链）相对琐碎，可以合并或精简。

4. **缺少对"不做什么"的系统性追问**：AI 大量追问了"做什么"和"怎么做"，但在"不做什么"（scope 裁剪）方面追问较少。Q3.4（Monorepo 不支持）和 Q2.14（断点恢复不支持）是我主动提出的，而非 AI 追问的。

5. **冷启动验证的时机**：按照课程要求，冷启动验证应在正式实现前完成。但实际开发中，这个环节容易被跳过（因为"已经花了这么多时间在 brainstorming 上，想赶快写代码"）。我需要在心态上把冷启动验证视为"设计审查"而非"额外负担"。

### 8.3 对 Brainstorming 技能的改进建议

1. **增加"scope 裁剪追问"**：在每轮结束时，AI 应主动追问"本轮的决策中，有没有什么应该明确排除的？"
2. **自动生成 SPEC 草稿**：在 brainstorming 结束后，AI 应将对话中的决策自动整理为 SPEC 的初步结构，减少人工整理工作。
3. **追问"测试策略"应更早出现**：Q3.12/Q3.13（测试策略）在第三轮才出现，但测试策略会影响实现方式，应该在第二轮就讨论。
