> 本文档记录阅读 SPEC.md v2.0 和 Plan.md v2.0 后发现的疑问与不一致之处。

## 1. SPEC.md 中引用的外部文档 §A.4(C) 是什么？

SPEC.md 多处引用 `§A.4(C)`，例如：

- §3.7 目标漂移防护："遵循 §A.4(C) 的'移除 LLM 后仍可单测'原则"
- §4.2 确定性："所有核心机制可用 mock LLM 确定性复现，测试不依赖网络与真实 LLM（§A.4(C)）"
- §7.5 重点维度选择："天然契合 §A.4(C) 的'移除 LLM 还能单测验证'的硬标准"

但 SPEC.md 全文没有定义 `§A.4(C)` 是什么。这是外部课程文档的引用吗？如果是，其关键约束或原则是什么？是否应在此文档中简要说明或提供链接？

## 2. 工具列表中缺少 Git 相关工具

SPEC.md §7.3 明确列出了该领域所需的工具，包括：

- **Git（只读）**：`git status`、`git diff`、`git log`
- **Git（写入，需审批）**：`git commit`、`git push`

但 SPEC.md §3.5.1 的工具列表（以及 §3.6 的系统提示中的工具表）只有 12 种工具，**不包含任何 Git 相关工具**。Plan.md T3.1 和 T3.2 也只实现了这 12 种工具。

**问题**：Git 操作（只读和写入）是否应该作为独立工具加入工具列表？还是通过 `run_command` 工具间接执行？如果通过 `run_command`，如何区分 Git 操作的只读和写入（审批）？

## 3. 路径检查的双重实现：T3.2（动作分发器） vs T4.2（路径围栏）

Plan.md T3.2（动作分发器）提到：
> "路径验证：所有文件操作路径必须相对路径或绝对路径在工作区内"

而 T4.2（路径围栏）是 L2 护栏层，同样做路径越界检查。

SPEC.md §5.2 数据流显示：护栏检查（L1-L5）在动作分发器之前执行。那么 T3.2 中的"路径验证"似乎是冗余的。

**问题**：这两层路径检查的分工是什么？是 T3.2 做基础校验（如路径格式合法性），T4.2 做安全边界检查（越界、敏感路径）？还是 T3.2 中的路径验证应该移除，统一由护栏层处理？

## 7. Dry-Run 模式的"确认后切换"逻辑在哪里实现？

SPEC.md §3.8.4 描述 Dry-Run 流程：
1. Agent 只执行只读操作，写操作仅记录到计划中
2. 展示计划摘要
3. **询问是否继续执行**
4. **确认后切换为正常模式执行**

但 Plan.md 中：
- T3.2 提到 "Dry-run 模式：写操作仅记录到计划，不实际执行"
- T3.6 提到 "支持 dry-run 模式（写操作仅记录不执行）"

**都没有提到"确认后切换为正常模式执行"的逻辑**。

**问题**：Dry-run 是只展示计划就结束，还是展示计划后等待确认再执行？如果后者，这个"确认后切换"的逻辑应该放在哪个 task 中？当前 Plan 中似乎没有对应的实现。
