# AI 开发实战指南：如何驾驭 AI 智能体完成真实软件项目

> 本指南提炼自 AI4SE 课程中 CodeHarness 项目的完整工程实践——从 3 轮 brainstorming（共 55 个设计问题）、到 SPEC 冻结、到 PLAN 拆分、到冷启动验证、到 subagent 驱动的实现。
>
> 核心命题：**当 AI 能完成大部分编码工作时，工程师的真正价值在哪里？** 答案不在"让 AI 替你写代码"，而在"你如何引导、治理、验证 AI 的输出"。

## 1. 核心理念：你不再只是写代码的人

在使用 AI 开发时，你的角色发生了根本转变：

| 传统角色 | AI 协作下的角色 |
|---------|---------------|
| 编码者（写每一行代码） | 架构师 + PM + Reviewer |
| 调试者（自己排查 bug） | 治理者（设计反馈闭环，让系统自修正） |
| 测试编写者 | 测试验收者（定义验收标准，验证结果） |
| 文档撰写者 | 规约编写者（把模糊需求变成可验证的 SPEC） |

**你的核心工作变成：**

1. **把模糊想法转化为清晰的规约（SPEC）**——AI 最怕的是不明确，最擅长的是执行明确指令。
2. **把大任务拆成小任务（PLAN）**——每个任务 2-5 分钟完成，颗粒度足够小，AI 一次会话能搞定。
3. **设计治理机制**——反馈信号、护栏、审批流程，确保 AI 不跑偏。
4. **阅读、评审、修正 AI 的产出**——你不需要写代码，但你必须能读懂代码。

> **一句话总结**：AI 是 CPU，你写的 harness（治理层）才是操作系统。你的价值在于工程判断力，而非打字速度。

## 2. 完整工作流：从想法到交付的七步法

```
┌──────────────────────────────────────────────────────────────────────┐
│  brainstorming  →  writing-plans  →  冷启动验证  →  subagent 开发     │
│      ↓                  ↓                ↓               ↓           │
│   SPEC.md           PLAN.md        SPEC_PROCESS.md    代码 + 测试     │
│                                                          ↓           │
│  finishing-branch  ←  code-review  ←  TDD（红→绿→重构）               │
└──────────────────────────────────────────────────────────────────────┘
```

**铁律：在 SPEC 和 PLAN 完成并通过冷启动验证之前，禁止写任何实现代码。**

这条铁律的背后逻辑是：你和 AI 在 brainstorming 阶段会沉淀大量共享的隐性上下文——你心里清楚但 SPEC 没写明的假设。这会让你严重高估 SPEC 的清晰度。跳过冷启动验证直接写代码，等于在沙地上盖楼。

## 3. 第一步：Brainstorming——与 AI 共同设计

### 3.1 启动方式

使用 Superpowers 的 `brainstorming` 技能启动你的 AI 智能体。它会主动追问"你究竟想做什么"，分块呈现设计供你逐步签字确认。

### 3.2 你的姿态：质询者，而非领受者

你不是在"领受指令"，而是在"质询"。任何含糊之处都应要求 AI 补足。好的 brainstorming 应该像一场攻防：

- **AI 提出方案** → 你追问"如果用户输入非法的怎么办？"
- **AI 补充细节** → 你追问"这个设计的性能瓶颈在哪？"
- **AI 给出答案** → 你追问"有没有更简单的方案？"

### 3.3 多轮 brainstorming 的节奏

一次 brainstorming 是不够的。CodeHarness 项目经历了 3 轮、共 55 个问题的深挖：

| 轮次 | 聚焦领域 | 典型问题 |
|------|---------|---------|
| **第一轮** | 项目定位与技术选型 | 目标用户是谁？用什么语言？对接哪些 LLM？ |
| **第二轮** | 核心机制细节 | 护栏的拦截顺序？HITL 状态机怎么设计？记忆存什么？ |
| **第三轮** | 工程实践与边界情况 | 系统提示怎么写？超大项目怎么处理？LLM 降级策略？ |

**每轮的重点**：
- 第一轮：定方向。回答"做什么"和"用什么做"。
- 第二轮：抠细节。把每个机制追问到代码级别——"这个 if 判断的条件是什么？"。
- 第三轮：封边界。把"用户输入为空怎么办""API 限流了怎么办""10 万文件的仓库怎么办"全部想清楚。

### 3.4 产出：SPEC.md

Brainstorming 的最终产出是一份完整的 `SPEC.md`，必须包含：

| 章节 | 核心问题 |
|------|---------|
| **问题陈述** | 要解决什么问题？目标用户是谁？为什么值得做？ |
| **用户故事** | 至少 5 个，遵循 INVEST 原则（独立、可协商、有价值、可估算、小、可测试） |
| **功能规约** | 按模块拆分，写明输入/行为/输出/边界条件/错误处理 |
| **非功能性需求** | 性能、安全（含凭据威胁模型）、可用性、可观测性 |
| **系统架构** | 组件图、数据流、外部依赖 |
| **数据模型** | 实体、字段、关系、约束——用 TypeScript 接口定义 |
| **领域与机制设计** | 该领域的反馈信号、危险动作、所需工具、记忆需求分别是什么？ |
| **凭据与分发设计** | Key 如何存储？如何分发给用户？首次运行体验如何？ |
| **技术选型与理由** | 为什么选这个语言？为什么选这个 LLM 供应商？ |
| **验收标准** | 每个功能"完成"的客观判定标准 |
| **风险与未决问题** | 你预见到的、可能让 AI 出问题的环节 |

**如果做 Coding Agent Harness 项目，SPEC 必须额外包含**：六维度（决策封装 / 工具分发 / 记忆 / 治理 / 反馈 / 配置）的设计方案，以及选哪个维度做深、为什么。

### 3.5 记录过程：SPEC_PROCESS.md

你需要记录 brainstorming 的过程，这不是形式主义——它帮助你反思：

- AI 追问了哪些好问题？哪些让你修正了原设想？
- 哪些建议是 AI 提出而你采纳的？哪些是你推翻的？为什么？
- 至少记录 3 轮关键迭代的对话节选与你的决策。

**如何写 SPEC_PROCESS.md**：

```
## 第一轮 brainstorming
- 日期：2026-07-07
- AI 提出的关键问题：……
- 我的回答与决策：……
- 本轮修正了哪些原设想：……

## 第二轮 brainstorming
- ……

## 第三轮 brainstorming
- ……

## 冷启动验证
- 第二个 AI 在哪里暂停并提问？暴露了哪些 SPEC 缺陷？
- 它做出了哪些与你原意不一致的解读？
- 你据此对 SPEC / PLAN 做了哪些修订？
```

## 4. 第二步：Writing Plans——把设计变成可执行计划

### 4.1 任务拆分原则

触发 `writing-plans` 技能，将设计分解为任务列表。每个任务必须满足：

- **颗粒度**：一个 subagent 在一次会话内能完成（约 2-5 分钟）。
- **包含目标**：这个任务要达成什么。
- **包含涉及文件**：明确要修改哪些文件。
- **包含验证步骤**：如何验证任务完成（含将要写的失败测试）。
- **标注依赖**：哪些任务必须先完成，哪些可以并行。

### 4.2 任务规划示例

```
T0.1 — 初始化项目骨架
  - 创建 package.json、tsconfig.json、jest.config.ts
  - 依赖：无
  - 验证：npm test 跑通，npx tsc --noEmit 通过

T1.1 — 工作区根目录检测 [P]  ← [P] 表示可并行
T1.2 — Shell 命令执行器 [P]
T1.3 — 文件操作工具（含原子性）[P]
T1.4 — 凭据存储 [P]

T2.1 — 配置加载器
  - 依赖：T1.1, T1.4
  - 验证：YAML 解析 + 深度合并 + 环境变量覆盖

T2.2 — 审计日志 [P]
T2.3 — 运行时日志 [P]

T5.1 — 护栏主逻辑
  - 依赖：T2.1, T2.2
  - 验证：guardrail("rm -rf /") → 被拦截，guardrail("npm test") → 通过
```

### 4.3 并行规划

利用 git worktree 机制，将无依赖的任务标记为可并行。例如：

```
Task 1: 搭建项目骨架（依赖：无）──┐
Task 2: 实现用户认证（依赖：Task 1）│ 可并行
Task 3: 实现日志模块（依赖：无）──┘
Task 4: 集成认证与日志（依赖：Task 2, Task 3）
```

### 4.4 产出：PLAN.md

一份好的 PLAN.md 让 AI 拿到后不需要再问你任何问题就能直接开工。格式要求：

- 每个 task 编号清晰（如 T0.1、T1.2）
- 每个 task 包含：目标、涉及文件、实现要点、验证步骤、依赖
- 可并行的 task 标注 `[P]`
- 按阶段分组（阶段 0：脚手架，阶段 1：基础设施，阶段 2：核心机制……）

## 5. 第三步：规约冷启动验证——用陌生 AI 检验你的设计

### 5.1 为什么必须做这一步

你和主 AI 在 brainstorming 阶段会沉淀大量**共享的隐性上下文**——你心里清楚但 SPEC 没写明的假设。这会让你严重高估 SPEC 的清晰度。

**一个全新的 AI 会在你未明文写下的每个假设处受阻——这些受阻之处，恰恰是 SPEC 质量最有价值的反馈信号。**

### 5.2 操作步骤

1. 换一个**不同类型**的 AI 智能体（如果你用 Claude Code 做开发，换用 Cursor 或 Codex 做验证）。
2. 启动**全新会话**，不导入任何先前会话或记忆。
3. **仅提供 SPEC.md + PLAN.md**，不补充任何口头解释。
4. 指定它从 PLAN 中选 1-2 个 task 自主推进。
5. 明确要求"遇到不确定之处即暂停询问，而非凭猜测继续"。

### 5.3 记录验证结果

在 `SPEC_PROCESS.md` 中记录：

- 第二个 AI 在哪里暂停并提问？暴露了哪些 SPEC 缺陷？
- 它做出了哪些与你原意不一致的解读？是 SPEC 写错还是它读错？
- 你据此对 SPEC / PLAN 做了哪些修订？给出修订前后的关键 diff。

## 6. 第四步：Subagent 驱动开发——让 AI 并行推进

### 6.1 工作区隔离

使用 git worktree 为每个独立功能创建隔离的工作区：

```bash
git worktree add ../feature-auth feature/auth
git worktree add ../feature-logging feature/logging
```

每个 worktree = 一个 PR，互不干扰。

### 6.2 派发子任务

每个 task 派一个**新鲜 subagent** 完成。关键原则：

- **一个 subagent 只做一件事**——不要让它同时改三个模块。
- **给足上下文**——在 prompt 中明确引用 SPEC 中的相关章节、PLAN 中的验证步骤。
- **设定明确的"完成"标准**——不是"实现了功能"，而是"测试通过 + 无 lint 错误"。

### 6.3 保持 PLAN.md 更新

每完成一个 task，立即在 PLAN.md 中标记完成并附上 commit hash。这是你跟踪进度的唯一真实来源。

## 7. 第五步：TDD 纪律——先红、再绿、再重构

### 7.1 TDD 三步法

这是**硬性要求**，不可跳过：

```
🔴 RED   → 先写一个失败的测试，运行它，确认它失败
🟢 GREEN → 写最少代码让测试通过，运行它，确认它通过
🔵 REFACTOR → 重构代码，保持测试绿色
```

### 7.2 为什么 AI 协作下 TDD 更重要

- 没有测试，你无法判断 AI 写的代码是否正确。
- 测试是**客观反馈信号**——它不依赖 AI 的"自我感觉"。
- 测试是**回归保护网**——后续 AI 修改代码时，测试会告诉你是否破坏了已有功能。

### 7.3 实操要点

- 让 AI **先写测试**，再让 AI 写实现。不要颠倒顺序。
- 如果 AI 写实现时跳过了测试，**打回去重做**。
- 你不需要亲自写测试代码，但你**必须理解测试在测什么**。

## 8. 第六步：两阶段评审——先合规、再质量

### 8.1 第一阶段：SPEC 合规检查

对照 SPEC 逐条检查：

- 输入/输出是否符合规约？
- 边界条件是否处理？
- 错误情况是否有恰当响应？

### 8.2 第二阶段：代码质量检查

- 代码是否清晰可读？
- 是否有明显的性能问题？
- 是否有安全漏洞？
- 凭据是否硬编码？（这是致命问题）

### 8.3 评审标准

- **Critical issue**：必须修复才能进入下一 task。
- **Minor issue**：记录但不阻塞，可在后续迭代中处理。

## 9. 第七步：完成分支——收尾与交付

### 9.1 Finishing a Branch

使用 `finishing-a-development-branch` 技能决定：

- 合并到主分支
- 创建 PR 等待人工审查
- 保留为独立分支
- 丢弃（如果方向错误）

### 9.2 最终交付物清单

| 文件 | 说明 |
|------|------|
| `SPEC.md` | 完整设计文档 |
| `PLAN.md` | 实现计划（每 task 标记完成状态） |
| `SPEC_PROCESS.md` | 设计过程记录 |
| 源代码 | 带完整 commit/PR 历史 |
| `README.md` | 项目简介、安装、运行、分发命令、安全说明 |
| `AGENT_LOG.md` | 开发过程日志 |
| CI 配置 | 一键运行测试，每次 push 自动触发 |
| `REFLECTION.md` | 1500-2500 字反思报告 |

## 10. 工程底线：凭据安全与分发

### 10.1 凭据安全（铁律）

> **Key 绝不硬编码进源码、绝不提交进 Git、绝不写入日志。**

**你必须做到：**

| 要求 | 说明 |
|------|------|
| 安全存储 | 操作系统钥匙串（Windows Credential Manager / macOS Keychain / Linux Secret Service），或加密文件 |
| 环境变量 | 只能通过 `.env` 文件加载（`.env` 本身不入 Git），不能命令行 `export`（会进 shell history） |
| 首次引导 | 程序首次运行应能引导用户安全录入 Key（隐藏输入） |
| 查看/更新/清除 | 查看状态时不得回显明文 |
| 威胁模型 | 在 SPEC 中明确：你的凭据方案防范什么、不防范什么 |

### 10.2 分发

让别人能真正用起来，任选一种或多种形态：

| 形态 | 要求 |
|------|------|
| **容器镜像** | `docker build` + `docker run` 一键启动，推送到公开 registry |
| **原生二进制** | 单文件可执行，说明目标平台与 CPU 架构 |
| **包管理器** | npm / PyPI / cargo / Homebrew，给出安装命令 |

README 中必须写清：获取方式、运行命令、**Key 如何在目标机器上安全配置**、已知限制。

## 11. 高级主题：构建你自己的 Coding Agent Harness

### 11.1 什么是 Harness

如果你在使用 AI 开发的过程中，想更进一步——**自己造一个 AI 智能体框架**，你需要理解 harness 的核心等式：

```
Agent = LLM + Harness
```

LLM 只是"决策引擎"，它的输出只是一段文本（"下一步应该做什么"）。Harness 是把这段文本变成实际行动的工程层：

```
┌──────────────────────────────────────────────────────────┐
│                      Harness                              │
│  ┌─────────────┐  ┌──────────────┐  ┌───────────────┐   │
│  │ 决策封装     │  │ 动作/工具     │  │ 上下文        │   │
│  │ 主循环       │  │ 读写文件      │  │ 记忆管理      │   │
│  │ 动作解析     │  │ 执行命令      │  │ Token 预算    │   │
│  │ 停机判断     │  │ 搜索代码      │  │ 上下文裁剪    │   │
│  └─────────────┘  └──────────────┘  └───────────────┘   │
│                                                          │
│  ┌─────────────┐  ┌──────────────┐  ┌───────────────┐   │
│  │ 治理护栏     │  │ 反馈闭环      │  │ 配置          │   │
│  │ L1 模式匹配  │  │ 测试运行      │  │ YAML 多层合并 │   │
│  │ L2 路径围栏  │  │ Lint 检查     │  │ 环境变量覆盖  │   │
│  │ L3 风险分级  │  │ 类型检查      │  │ CLI 参数      │   │
│  │ L4 HITL 审批 │  │ 构建验证      │  └───────────────┘   │
│  │ L5 审计日志  │  │ Diff 追踪     │                      │
│  └─────────────┘  └──────────────┘                      │
└──────────────────────────────────────────────────────────┘
```

### 11.2 六维度设计

一个完整的 harness 需要覆盖六个维度：

| 维度 | 核心问题 | Coding 领域的实现 |
|------|---------|-----------------|
| **决策封装** | 如何组织上下文、调用 LLM、解析动作、判断停机？ | Agent 主循环 + 动作解析器 + 偏离检测 |
| **动作/工具** | Agent 能执行哪些操作？ | 12 种动作类型：读写文件、执行 shell、搜索、Git 操作等 |
| **客观反馈信号** | 什么信号告诉 Agent"你做对了"？ | 运行测试/lint/类型检查/构建，结果客观确定 |
| **危险动作** | 什么操作必须人工审批？ | 五层护栏：L1 模式匹配 → L2 路径围栏 → L3 风险分级 → L4 HITL → L5 审计 |
| **记忆** | 跨会话需要记住什么？ | 项目约定、历史决策、代码库知识、用户偏好 |
| **配置** | 用户如何约束 Agent 行为？ | YAML 多层配置 + 环境变量 + CLI 参数覆盖 |

### 11.3 核心纪律：机制必须是代码，不是提示词

这是区分"真正实现了 harness"和"只是给 LLM 写了几句提示词"的硬标准：

**❌ 提示词版（不算实现）：**
```
系统提示：不可以执行删除数据库的命令。
```
是否执行取决于 LLM 是否"听话"，每次结果可能不同，你无法写测试验证。

**✅ 代码版（才算实现）：**
```python
def guardrail(action):
    dangerous_patterns = ["rm -rf", "DROP TABLE", "DELETE FROM"]
    for pattern in dangerous_patterns:
        if pattern in action.command:
            return {"blocked": True, "reason": f"危险命令: {pattern}"}
    return {"blocked": False}
```
你可以在没有 LLM 的情况下写单元测试：传入 `Action(command="rm -rf /")`，断言被拦截，每次都成立。

### 11.4 可测试性判据

> **移除真实 LLM 后，你的每个核心机制还能用 mock/stub LLM 做确定性单元测试验证吗？**

如果可以 → 你编码实现了机制。
如果不可以（因为依赖 LLM 的"智能"） → 你只是写了提示词。

**以护栏为例的单元测试设计**：

```typescript
describe('guardrail', () => {
  it('正常命令应直接通过', () => {
    const result = guardrail('npm test', workspaceRoot);
    expect(result.passed).toBe(true);
    expect(result.riskLevel).toBe('safe');
  });

  it('危险命令应触发审批', () => {
    const result = guardrail('rm -rf ./node_modules', workspaceRoot);
    expect(result.passed).toBe(false);
    expect(result.riskLevel).toBe('dangerous');
    expect(result.requiresApproval).toBe(true);
  });

  it('工作区外路径应无条件拒绝', () => {
    const result = guardrail('cat /etc/passwd', workspaceRoot);
    expect(result.passed).toBe(false);
    expect(result.riskLevel).toBe('fatal');
    expect(result.requiresApproval).toBe(false); // 不审批，直接拒绝
  });
});
```

这些测试不依赖真实 LLM，传入构造的输入，断言确定性输出。每次运行结果一致。

### 11.5 实现边界

| 允许 | 不允许 |
|------|--------|
| 使用 LLM 供应商的单次对话补全 API | 使用现成 agent 框架的高层循环（如 LangChain AgentExecutor） |
| 使用 HTTP 库、向量库、解析库等底层零件 | 把项目建在 AutoGen、CrewAI 等编排框架之上 |
| 自己实现 agent 主循环 | 用某编码智能体 SDK 自带的 agent runner |
| 自己实现可注入 mock 的 LLM 抽象层 | 让框架的 agent loop 充当你的产物功能 |

> **简单说：你可以用螺丝刀和木板，但不能买一个现成的柜子然后说"这是我做的"。**

### 11.6 选一个维度做深

六个维度都要有最低可运行实现，但**选一个做深**。CodeHarness 选了**治理（护栏）**，因为：

1. 危险命令检测、路径越界检查、审批状态机、超时控制——全都是确定性的代码逻辑
2. 五层护栏架构（L1-L5），每层都有明确的代码实现和测试
3. 机制演示最容易做：三个演示中有两个直接展示治理能力
4. 与其他维度自然关联：拦截结果作为反馈回灌、护栏规则可配置、动作执行前经过护栏

**治理深度的五层设计**：

| 层级 | 机制 | 实现要点 |
|------|------|---------|
| L1 | 模式匹配 | 8 类危险模式的正则检测（文件破坏、Git 破坏、对外发布、任意代码执行、数据库破坏等） |
| L2 | 路径围栏 | 工作区根目录边界检测，敏感路径额外检测 |
| L3 | 风险分级 | 四级风险（safe/caution/dangerous/fatal），决定审批策略 |
| L4 | HITL 状态机 | 完整状态图：审批/拒绝/超时/会话白名单（按危险类别记忆） |
| L5 | 审计日志 | 结构化 JSONL 记录所有动作、护栏结果、审批结果 |

### 11.7 反馈闭环架构

反馈闭环是 harness 的"自我修正"能力。核心设计：

```
Agent 执行动作 → 反馈收集器运行
                    ├── Build（失败则跳过后续，实现依赖短路）
                    ├── Lint
                    ├── TypeCheck
                    ├── Test
                    └── Diff
                         ↓
                    一次性汇总回灌到上下文
```

**关键设计决策**：
- **依赖短路**：构建失败 → 跳过测试，避免无意义的浪费
- **一次性回灌**：所有反馈结果汇总后一次性注入，而非逐条注入
- **自动修正上限**：`auto_fix_rounds` 硬上限（默认 3 轮），防止无限修正循环

## 12. 反思与批判：AI 开发方法论的边界

### 12.1 这套方法论假设了什么

- **假设 SPEC 可以预先写清楚**：但对于探索性项目，需求本身就是模糊的，过早写死 SPEC 可能是束缚。
- **假设 TDD 总是最优**：但对于原型探索阶段，快速试错可能比严格 TDD 更高效。
- **假设 subagent 可以独立工作**：但复杂功能往往需要全局理解，独立 subagent 容易做出局部最优但全局不协调的决策。
- **假设 AI 的"理解"是稳定的**：但相同的 prompt 在不同时间可能得到不同结果。

### 12.2 你应该形成自己的判断

在使用这套方法论完成项目后，你应该能回答：

1. 哪些环节 AI 确实极大提升了效率？哪些环节 AI 反而增加了沟通成本？
2. TDD 强制在 AI 协作下是阻碍还是放大器？
3. Subagent 能自主运行多久而不偏离主题？
4. 什么样的 task 颗粒度最优？太大 AI 跑偏，太小沟通开销超过编码收益。
5. SPEC 里一个模糊的表述，如何导致 AI 在实现时完全跑偏？
6. 凭据安全和分发这两条工程要求，迫使你想清楚了哪些原本会忽略的问题？

### 12.3 最终的批判

> **Superpowers 给你的是"流程脚手架"——它守住了 TDD、评审、计划这些在 AI 协作中容易松懈的纪律。但它不能替你回答"做什么"和"做对了吗"。**
>
> 如果你只是把题目交给 AI、把结果原封不动交上来，你浪费了这次学习机会。真正的成长在于：你能解释这套方法论为何这样设计、它在何处仍有不足、你如何引导它做对，并对"一个可靠的系统到底需要哪些工程"给出第一手的回答。

```
1. brainstorming    → 产出 SPEC.md（不要写代码！）

```

### 你的职责清单

- [ ] 写出清晰的 SPEC（AI 能读懂、人类也能读懂）
- [ ] 拆出粒度合理的 PLAN（每个 task 2-5 分钟）
- [ ] 换一个 AI 做冷启动验证
- [ ] 为每个 task 派发新鲜 subagent
- [ ] 坚持 TDD（先红后绿再重构）
