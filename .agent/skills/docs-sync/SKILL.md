# 文档同步 Skill

> **角色**：当新增或修改用户级别功能时，自动同步更新 PLAN.md、SPEC.md 和 README.md。确保文档始终与代码实现一致。

---

## 触发条件

以下场景触发此 Skill：

1. 新增 CLI 命令或子命令
2. 新增配置项或配置 Schema 变更
3. 新增 LLM 提供商或适配器
4. 新增交互模式（如对话模式、会话管理）
5. 修改用户可见的行为或输出格式
6. 修改退出码或错误处理逻辑

---

## 同步规则

### 规则 1：PLAN.md 更新

**何时更新**：所有用户功能变更都需要在 PLAN.md 中记录。

**更新位置**：在 PLAN.md 末尾当前最新阶段追加新 Task。

**格式规范**：

```markdown
### ✅ T<阶段>.<序号> — <任务名称>

- **目标**：<一句话描述>
- **涉及文件**：`<文件列表>`
- **实现要点**：
  1. <要点1>
  2. <要点2>
- **验证**：<验证方式>
```

**如果是已有阶段的补充**：在对应阶段末尾追加 Task，序号递增。

**如果是全新阶段**：新建 `## 阶段 N：<阶段名称>` 区块，包含：
- 背景说明（`> **背景**：...`）
- 各 Task 条目
- 变更汇总表（新增模块表格 + 更新模块表格）
- 更新文档版本号

**版本号规范**：PLAN.md 版本号格式为 `v<主版本>.<次版本>`，功能增量递增次版本号。

---

### 规则 2：SPEC.md 更新

**何时更新**：功能涉及新的设计决策、接口变更、或用户可见行为变更时需要更新。

**更新位置**：在对应章节下新增子章节，或新建章节。

**格式规范**：

```markdown
### 3.X <功能名称>

**动机**：<为什么需要这个功能>

**3.X.1 基本用法**

```bash
codeharness <命令示例>
```

**3.X.2 配置格式**（如有）

```yaml
<配置示例>
```

**3.X.3 <子功能说明>**（如有）
```

**变更历史更新**：在 `## 12. 变更历史` 表格中追加新行：

```markdown
| <版本> | <日期> | P1 功能增量 | <变更说明> | <受影响章节> |
```

**版本号更新**：文档底部版本号同步更新。

---

### 规则 3：README.md 更新

**何时更新**：所有用户可见功能都需要在 README 中体现。

**更新内容**：

1. **命令一览表**：新增命令需添加到"命令一览"表格
2. **配置示例**：新增配置项需更新 YAML 示例
3. **快速开始**：重大功能需更新快速开始示例
4. **架构图**：新增模块需更新架构图
5. **模式说明**：新增运行模式需更新对应表格

**格式**：README.md 使用中文，表格对齐，代码块带语言标注。

## 执行流程

### 步骤 1：评估变更范围

在开始实现前，先回答以下问题：

| 问题 | 是/否 | 影响的文档 |
|------|-------|-----------|
| 是否新增 CLI 命令？ | | PLAN + SPEC + README |
| 是否新增配置项？ | | PLAN + SPEC + README |
| 是否新增模块/文件？ | | PLAN + SPEC |
| 是否改变用户交互方式？ | | SPEC + README |
| 是否改变输出格式？ | | SPEC + README |
| 是否改变退出码？ | | SPEC + README |

### 步骤 2：实现功能

按照 PLAN.md 中对应 Task 的实现要点执行。

### 步骤 3：同步文档

实现完成后，按下方清单逐项更新：

#### PLAN.md 更新清单

- [ ] 新增 Task 条目（含目标、涉及文件、实现要点、验证）
- [ ] 更新变更汇总表（新增模块 + 更新模块）
- [ ] 更新版本号（次版本 +1）
- [ ] 更新最后更新日期
- [ ] 更新总计 Tasks 数量

#### SPEC.md 更新清单

- [ ] 新增功能章节（含动机、用法、配置格式）
- [ ] 更新变更历史表格
- [ ] 更新版本号
- [ ] 更新目录（如需要）

#### README.md 更新清单

- [ ] 更新命令一览表
- [ ] 更新快速开始示例（如需要）
- [ ] 更新配置示例 YAML
- [ ] 更新架构图（如需要）
- [ ] 更新模式/提供商/退出码表格

### 步骤 4：验证

- [ ] 运行 `npm test` 确认所有测试通过
- [ ] 运行 `npx tsc --noEmit` 确认编译通过
- [ ] 检查 PLAN.md 中 Task 标题是否标记为 `✅`
- [ ] 检查 SPEC.md 变更历史是否记录
- [ ] 检查 README.md 命令示例是否与实际一致

## 文档格式速查

### PLAN.md Task 模板

```markdown
### ✅ T12.1 — 功能名称

- **目标**：一句话描述
- **涉及文件**：`src/xxx.ts`, `tests/xxx.test.ts`
- **实现要点**：
  1. 要点1
  2. 要点2
- **验证**：验证方式
```

### SPEC.md 章节模板

```markdown
### 3.X 功能名称

**动机**：为什么需要

**3.X.1 基本用法**

```bash
命令示例
```

**3.X.2 配置**

```yaml
配置示例
```
```

### 变更汇总表模板

```markdown
| 新增模块 | 对应 Task | 说明 |
|---------|----------|------|
| 模块名称 | T12.X | 功能说明 |

| 更新模块 | 对应 Task | 变更内容 |
|---------|----------|---------|
| 模块名称 | T12.X | 变更说明 |
```

## 常见场景示例

### 场景 A：新增 CLI 命令

```bash
# 用户说：添加一个 codeharness status 命令
```

1. PLAN.md 对应阶段追加 T12.X Task
2. SPEC.md §3.11 追加子章节说明命令用法
3. README.md 命令一览表追加新行

### 场景 B：新增配置项

```bash
# 用户说：添加 max_tokens 配置项
```

1. PLAN.md 追加 Task
2. SPEC.md §6.3 更新配置 Schema
3. README.md 配置示例 YAML 追加新字段

### 场景 C：新增 LLM 提供商

```bash
# 用户说：支持 Gemini 提供商
```

1. PLAN.md 追加 Task（含涉及文件列表）
2. SPEC.md §3.12 追加提供商类型说明
3. README.md 提供商表格追加新行

## 禁止事项

1. **禁止实现功能后不更新文档**：代码和文档必须同步
2. **禁止更新文档但不更新版本号**：每次变更必须递增版本号
3. **禁止在未确认功能规格前更新 SPEC.md**：SPEC 是设计文档，先确认再更新
4. **禁止 README.md 包含过期示例**：命令示例必须与实际代码行为一致
5. **禁止跳过验证步骤**：文档更新后必须运行测试确认无回归
